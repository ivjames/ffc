Art & UI Style Guide

A print reference for the artist theming the app. Every screen, every button, every interface element — with the app's real materials rendered so you can match or deliberately diverge from them.

14 skins Light + Dark Per-course tint 28 screens & overlays Full icon-slot set

How to use this guide

  1. Read §1 (the theming contract) first. Art plugs into three independent axes — skin × light/dark × course color. Respect the hooks and your art drops straight in.
  2. §2–§4 are the kit: palette, type, motion, and the shared materials (buttons, surfaces, tiles, pucks, tags) rendered as real specimens.
  3. §5 is the coverage checklist. Walk it screen by screen and tick every element — that's how we guarantee nothing's missed.
  4. §6 is the icon-slot set (every emoji is a placeholder for a designed icon). §7 is delivery specs. §8 is the one-page master checklist.
  5. Validate live: drop any asset into the running app at /style and cycle all skins × light/dark. Companion text spec: docs/art-spec.md.

Concrete sizes, colors, radii and fonts below describe what the app ships today — treat them as “e.g.” and a starting palette, not a mandate. What is fixed is the structure: the theming hooks, the element slots, and the functional constraints (tap-target sizes, safe areas, contrast). Those are flagged [constraint].

01 The theming contract — read first

The app composes three independent visual axes on the page. Art must survive every combination, or ship one variant per relevant axis. A skin restyles only the shared material classes, so changing it re-skins every screen at once.

How art plugs in. Most elements are pure CSS today. To theme with art, either (a) replace a material with an SVG/PNG/WEBP background, or (b) supply per-skin and/or per-mode variants. Prefer inline SVG that inherits currentColor and reads --accent/--tile-accent where a color must track the course. Provide a light + dark treatment unless one asset clears contrast on both grounds.
Emoji are placeholders, not a standard option. Every glyph you see (course markers, controls, activity icons) is a stand-in. Each real theme ships its own image icon set — SVG preferred, WEBP/PNG for illustrative icons. A dedicated “Emoji” theme is the only exception. Treat every emoji in §5–§6 as a slot that needs designed art.

03 Typography & motion

The app ships system font stacks today for zero-download performance, but typography is one of the biggest levers a theme has — you're expected to bring your own faces. Fill both roles; the families are open.

Two type roles

  • UI / body face — screens, buttons, copy. Current default: system sans (a chunky display weight for headings suits the arcade tone).
  • Mono / “arcade” face — player tags, winner names, the scorecard tag column, rank numerals. Current default: system mono + .font-arcade (letter-spacing 0.15em). Keep it monospaced enough to align in columns.

Scale (px, current): 48 · 36 · 30 · 24 · 20 · 18 · 16 (body) · 14 · 12 (eyebrow) · 11/10/9.5 (micro). Re-scale freely; preserve the hierarchy.

Weights: display black (900), body bold (700), eyebrows uppercase + wide tracking.

Custom fonts must be self-hosted / bundled — the app avoids external font CDNs. Ship the files with the theme.

Layout & metrics

Column & radii

Column: max-w-md = 448px portrait, centered, 16px content padding. Compose art to a ~448px column.

Radii (current example): 8 (lg) · 12 (xl) · 16 (2xl) · 24 (3xl) · full. A theme may pick its own corner language (sharp, pill, etc.).

Constraints — honor these

[constraint] Safe areas: body pads env(safe-area-inset-*); the top bar pins below the notch. Keep art clear of notch / home-indicator zones.

[constraint] Tap targets: interactive elements ≥ ~44px hit area — current keys 56×56, back 40×40, control pills 36×36, buttons ~52 tall. Style freely but don't shrink below a comfortable touch target.

07 Deliverables & conventions

Format

SVG preferred for UI markers — single-color/duotone, fill="currentColor" where it should track text, reading var(--accent) / var(--tile-accent) where it must track the course. WEBP (or PNG) for richer illustrative icons and photographic art. Raster app icons at @1× / @2× / @3×. (Emoji only for an optional “Emoji” theme.)

Variants

Ship light + dark whenever one asset can't clear contrast on both grounds. If an asset is skin-specific (e.g. a candy-only gloss), name the skin; otherwise design skin-agnostic so it inherits the skin's material.

Sizing & contrast [constraint]

Author on the pixel sizes in §3 / §5–§6; keep strokes optically consistent. Contrast ≥ 4.5:1 (normal) / 3:1 (large ≥24px or bold ≥18.66px, non-text UI) — hard requirement regardless of aesthetics. Tags on bright accents = known tight spot.

Where assets live

App icons → public/icons/ · maps → public/maps/ · per-theme art → public/themes/<skin>/ · new icon sets under public/ (or inline SVG in components). Materials/skins are CSS in src/index.css under :root[data-template='…'].

Naming

element[-variant][-skin][-mode]@scale.ext
e.g. puck-dragon.svg · card-castle.webp · app-icon-512.png · wheel-pointer.svg

Validate

Open the app at /style (the live guide), cycle all 14 skins × light/dark, and confirm the asset reads and holds through its animation. Keep the companion docs/art-spec.md in sync when elements change.
